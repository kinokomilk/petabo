# petabo 検証手順（TESTING）

各フェーズ完了時に、機能・UI・セキュリティ・性能を確認する。すべて満たすまで「完了」にしない。問題や判断に迷う点があれば、勝手に直さずユーザーに報告・相談する。

---

## 0. 自動テストの方針

- ユニット/結合：**Vitest**（Workers は `@cloudflare/vitest-pool-workers` 等で D1 を含めて検証）。
- E2E/UI：**Playwright**（モバイルビューポートで主要フローを自動操作）。
- D1 はローカル（miniflare）で再現し、マイグレーション適用→シードでテスト。
- CI で `typecheck / test / build` を回す。

## 1. 機能テスト（想定通りに動くか）

### 認証
- [ ] フォールバック登録/ログイン、誤パスワードは拒否、未認証は 401。
- [ ] セッションの発行・失効・期限切れ挙動。
- [ ] （Phase 2）LINE ログインでユーザー作成/再ログイン、`userId` 重複時に同一ユーザーへ紐付く。IDトークン検証（署名/aud/nonce/exp）と friendship status による `line_followed` 同期。

### タスク/チェックリスト
- [ ] todo の CRUD、担当・期限・タグ・状態の更新が反映。
- [ ] フィルタ（status/assignee/tag）が正しく絞り込む。
- [ ] チェックリスト：項目の追加（Enter 連続・改行一括）、done 切替、進捗 n/m とバーが一致、並び順 position。
- [ ] クイック追加：入力→即追加、担当/カテゴリの**前回値引き継ぎ**。
- [ ] 「保存して続けて追加」で担当/期限/カテゴリが保持される。
- [ ] コメントの追加・表示。
- [ ] in-app リマインダー：overdue/間近の判定と色表示。

### LINE（Phase 2/3）
- [ ] webhook 署名検証（不正署名は拒否、生ボディ基準）。
- [ ] follow で userId 取得、連携確立。unfollow で push 抑止。
- [ ] Cron→担当者 push、`todo_reminders` で重複送信なし、期限変更で再送、静かな時間帯は送らない。removed member は担当者として残っていても通知しない。
- [ ] コマンド `一覧`/`追加`/完了ボタン(postback) が正しいタスクに作用（番号ズレが無いこと）。
- [ ] Flex 表示・リッチメニュー・LIFF 起動とプロフィール取得。

## 2. UI 検証（想定通りの見た目・操作か）

実装済みのデザイントークンと主要画面を基準に確認する。

- [ ] **モバイルファースト**：iPhone 幅（375–430px）で破綻なし。タップ領域・スクロール・固定要素を確認。
- [ ] 色アバター（人）・タスク行・**消し込み（取り消し線）**・**公開範囲の目印（鍵/じぶん）**が再現されている。
- [ ] 配色（オレンジ系：accent #FF7A4D / bg #FBF6EF、期限の赤/オレンジ）・フォント（Zen Maru Gothic 見出し / Noto Sans JP 本文）。
- [ ] フォント読込失敗時にシステム丸ゴシックへフォールバックして崩れない。
- [ ] アクセシビリティ：フォーカス可視、`prefers-reduced-motion` で過剰なアニメ抑制、コントラスト。
- [ ] Playwright でスクリーンショットを撮り、主要画面の崩れを目視確認。

## 3. セキュリティ確認

- [ ] LINE webhook の**署名検証**（HMAC-SHA256, 生ボディ）。検証前に副作用を起こさない。
- [ ] LINE IDトークン/アクセストークンの検証（署名・aud・iss・exp）。`userId` を検証前に信用しない。LINE Login後の通知可否は `bot_prompt` だけに依存しない。
- [ ] 秘密情報（チャンネルシークレット/トークン）は Workers Secrets。リポジトリ/クライアントに出さない。
- [ ] パスワードは PBKDF2＋ソルト。タイミング安全比較。総当り対策（緩いレート制限でよい）。
- [ ] 認可：他人のタスク/コメントを ID 直打ちで操作できないこと（共有スペース内の権限境界を定義し検証）。
- [ ] **非公開（private）タスクの隔離**：作成者以外には一覧・個別取得（ID 直打ち）・通知のいずれにも現れないこと。オーナーでも見えないこと。共有↔非公開の切替が即座に反映されること。
- [ ] 入力検証・SQL は**プレースホルダ必須**（文字列結合禁止）。XSS：React は基本安全だが `dangererouslySetInnerHTML` 不使用、LINE/外部由来テキストのエスケープ。
- [ ] CORS/Cookie：本番オリジンを限定。Cookie 採用時は HttpOnly/Secure/SameSite。
- [ ] セッショントークンは十分なエントロピー、失効可能。ログに秘密やトークンを残さない。
- [ ] LIFF/OAuth の state/nonce による CSRF/リプレイ対策。

## 4. 性能確認

- [ ] webhook は素早く 200 を返す（重い処理は `ctx.waitUntil()`）。LINE のリトライを誘発しない。
- [ ] D1 クエリに必要なインデックス（status/assignee/todo_id 等）。N+1 を避け、hydrate をまとめる。
- [ ] 一覧のページング/件数上限を考慮（家族規模では軽いが、無制限取得にしない）。
- [ ] Cron の処理時間と Worker CPU 制限内に収まること。対象タスクのみ抽出（全件走査を避ける）。
- [ ] 無料枠の範囲（Workers 10万req/日、D1 500万read/月、LINE 200通/月）に収まる設計であることを確認。
- [ ] フロント：初回ロード/バンドルサイズ、PWA のオフライン挙動、画像/フォントの読み込み。

## 5. 完了の定義（Definition of Done）

- 上記チェックを満たす／自動テストが緑／`build` 成功／デプロイ確認。
- 既知の制約・未対応・判断保留点を簡潔にまとめてユーザーへ共有。
- 仕様の解釈で迷った点は**勝手に確定せず質問**し、合意後に反映。
